# Scaffold vs. Original Work

This document records what came from the Google AI Studio template vs. what was designed and built from scratch, so the distinction can be stated accurately in interviews.

## What is original work

Everything in `backend/` is entirely original — the AI Studio template is JavaScript-only and generates no Python code whatsoever. The frontend components are also original designs built on top of the scaffold toolchain.

### backend/agent/core.py — Original Design

**Key decisions made here:**
- The flat `AVAILABLE_TOOLS` dict (31 entries) as a single dispatch table. This is a deliberate choice over a class-based plugin registry — functions are directly importable and the dict is the only source of truth for what the agent can invoke.
- The **5-iteration tool resolution loop** (`while loop_count < 5`): allows chained tool calls within a single turn (e.g., "generate morning briefing" internally needs LeetCode + Gmail + GitHub). The loop terminates early if no tool calls are detected.
- **Dual-path tool call parsing**: native Groq `function_calls` API response is primary; a regex tag parser (`<function=name>{args}</function>`) is a secondary fallback for models that output tool calls as text. Both paths feed the same unified `run_tool()` coroutine.
- **Model routing**: `llama-3.2-11b-vision-preview` for requests with images + visual keywords; `llama-3.3-70b-versatile` for everything else. Vision models are excluded from tool calls in the same turn (Groq API limitation — handled by setting `tools_for_turn = None` for vision calls).
- **Session cache** (`_session_cache`): a bounded in-process LRU (128 entries max) that sits in front of the Redis + Supabase history chain. This is an intentional L0 cache to skip Redis on consecutive turns from the same active user.
- **Selective result caching**: tool results are only cached when the result string does NOT start with `"Error:"`. This prevents a transient failure (e.g., Spotify 401) from being served as a cached "answer" for the next 5 minutes.

### backend/agent/memory.py — Original Design

**Key decisions made here:**
- **Three-layer history retrieval**: in-process session cache (L0) → Redis semantic cache (L1, 30-min TTL) → Supabase Postgres (L2, cold start). The layers were added incrementally to solve real observed latency.
- **Double-checked locking on Supabase init** (`_init_lock`): prevents multiple concurrent coroutines from all trying to initialize the Supabase client simultaneously on first request.
- **10-second init timeout** (`asyncio.wait_for(..., timeout=12.0)`): prevents the entire FastAPI startup from blocking if Supabase is unreachable.
- **Atomic upsert for `skill_mastery`**: uses Supabase's upsert with `on_conflict` to avoid 0-row errors when a skill category is first seen.
- **LeetCode streak calculation**: implemented manually in Python (not via a Supabase function) — counts backwards from today checking `completed=True`, handles the edge case where today is not yet marked done.

### backend/agent/prompt.py — Original Design

- **Dual-mode prompt switching**: JARVIS mode (default) and Batman/Alfred mode, toggled by a runtime flag in the Intelligence Hub (`batman_mode`). The same conversation history is shared across both modes.
- JARVIS system prompt includes explicit rules for: zero-emoji output, tool integrity (no hallucinated filenames), stress detection protocol (skip pleasantries when `is_stressed=True`), mobile sync mandate (must call tool if claiming to send to phone), and visual audit instructions.

### backend/voice/stt.py — Original Design

**The self-healing fallback logic is the most technically defensible part of this file:**
- `_choose_provider()`: if `STT_PROVIDER` env var is not set, defaults to Groq when a key is available, otherwise local. This means the system works out of the box in three modes: cloud-only, local-only, or automatic.
- **Bidirectional fallback**: Groq primary → local fallback (cloud mode), OR local primary → Groq fallback (local mode). The fallback chain runs in both directions depending on which provider is configured as primary.
- **CUDA self-heal** (lines 35–41): on Windows, `faster-whisper` sometimes detects a CUDA device but fails to use it (driver mismatch). The inner try/except silently retries with `device="cpu", compute_type="int8"` — no user-visible error.
- Audio format handling: `_suffix_from_mime()` maps browser MIME types (audio/webm, audio/ogg, audio/mp4, etc.) to file extensions for the Whisper temp-file approach. This was added after observing that browsers report different MIME types depending on OS and browser version.

### backend/voice/tts.py — Original Design

- **Pipeline-overlapped synthesis** (lines 235–251): uses `asyncio.create_task()` to start synthesizing chunk N+1 while chunk N is being sent to the client. This overlaps network/synthesis latency with client delivery time.
- **Per-chunk Groq fallback**: does NOT disable Groq session-wide after a single chunk failure. Each chunk independently attempts Groq then falls back to edge-tts. Rationale: the first chunk in a cold session frequently fails due to TLS handshake latency; later chunks succeed. Disabling Groq globally after one failure would degrade voice quality for the rest of the session.
- **Cold-start retry**: one extra edge-tts retry for early chunks (documented in comments: DNS/TLS cost on the first sentence made it sound like JARVIS only spoke the last sentence).
- **TTS warm-up** (`warm_up_tts`): runs at startup via `asyncio.create_task()` (non-blocking). Sends a throwaway "Online." synthesis to pre-open connections on both Groq and edge-tts. Without this, the first real reply of a session drops its opening sentence(s) to a cold-start timeout.
- **`clean_text()`**: strips markdown heading markers only at line-start (`^#+`) to preserve inline `#` characters like `C#`. A naive `re.sub(r'#+', '', text)` approach would silently mangle code examples spoken aloud.

### backend/voice/tts_engine/ — Original Design

A second, more modular TTS subsystem (separate from voice/tts.py) used by the `/api/tts` REST endpoint:
- **Provider abstraction** (`interface.py`: `TTSProvider` ABC): Kokoro (local ONNX), edge-tts, OpenAI, ElevenLabs all implement the same `generate_speech` / `stream_speech` interface.
- **File-based audio cache** (`cache.py`): SHA-256 keyed by `(text, provider, voice, speed, format)` with whitespace/casing normalization so "Hello World" and "hello world" share a cache entry.
- **Format transcoding** (`service.py`): pydub-based WAV↔MP3 conversion with graceful fallback (returns native format if ffmpeg is absent rather than crashing).

### backend/ws_neural.py — Original Design

- **Sentence-level TTS buffering** (three modes: `sentence`, `entire`, `chunk`): the `sentence` mode uses punctuation detection + a 15-token comma-pause threshold + a 25-token run-on fallback. This required iteration — the original "chunk" mode (split every 4-6 tokens) caused a stuttering voice effect on punctuation-heavy output.
- **Turn cancellation** (`cancel_voice_task()`): when a new voice input arrives mid-turn, the in-flight `run_agent` task is cancelled. The `finally` block in `run_agent` always sends `TURN_COMPLETE` with `interrupted=True` on cancel, so the frontend's browser-TTS fallback does not fire on the half-formed fragment (which would cause the voice to "restart mid-paragraph").
- **60-second WebSocket keepalive**: `asyncio.wait_for(websocket.receive(), timeout=60)` sends a PING if no message arrives in 60 seconds, preventing silent dead connections from load balancers.
- **Batman mode** (`BATMAN_TRIGGER`): short-circuit before hitting the LLM — phrase matching triggers a local reply and updates the Intelligence Hub flag, changing the system prompt for all subsequent turns without restarting anything.

### backend/ws_hub.py — Original Design

- The hub broadcasts a `SYSTEM_PULSE` packet every 5 seconds containing live CPU/RAM/disk/battery (via psutil), the current Intelligence Hub state (LeetCode, GitHub, Spotify, Gmail), focus session data, and drained proactive alerts.
- **Proactive trigger drain**: the sender loop iterates `proactive_triggers` dict, pops each key, and sends them as a list in the pulse packet. Triggers are injected by the calendar sentinel (upcoming meetings within 15 minutes) and the HMAC-signed edge endpoint.
- **Sender task watchdog** (`_on_sender_done` callback): if the background sender task crashes unexpectedly (e.g. a malformed trigger), the WebSocket is proactively closed so the receiver loop doesn't idle silently with no pulses.

### backend/services/cache_service.py — Original Design

- **Intelligence Hub**: a single thread-safe `dict` as the runtime source of truth for all dashboard state (Gmail unread count, LeetCode stats, GitHub pulse, Spotify track, focus session, etc.).
- **Atomic disk write**: uses `tempfile.NamedTemporaryFile` + `os.replace()` to ensure the sync file is never partially written — crash during write leaves the previous version intact.
- **Log rotation**: `communication.log` rotates at 5 MB to prevent unbounded growth in long-running sessions.

### backend/services/redis_service.py — Original Design

- **Private network enforcement**: rejects any Redis `REDIS_HOST` that is not RFC1918 (`127.0.0.1`, `192.168.*`, `10.*`, `172.16–31.*`). If a non-private host is configured (accidentally public), the `RedisCache` initializes with `self.client = None` and all cache operations silently no-op. This is a security decision: the Redis instance should never be internet-accessible.
- **Namespaced SHA-256 keys**: `jarvis:{namespace}:{sha256(content)}` — tool results are under `jarvis:tool:{tool_name}:{sha256(args)}`, semantic history under `jarvis:semantic:{sha256(query)}`. Hashing prevents key length overflow for large argument payloads.

### backend/celery_app.py + tasks.py — Original Design

- **Graceful eager fallback** (`task_always_eager=True`): if Redis is unreachable at startup, Celery switches to synchronous in-process execution so the app still works without a message broker. The task API is unchanged — callers use `.delay()` regardless.
- **STT via Celery** (`process_stt_task`): runs `transcribe_audio()` in a fresh `asyncio.new_event_loop()` per call, because Celery workers are synchronous threads that do not have an event loop. The explicit `loop.close()` in `finally` prevents loop leak.
- **HMAC-signed edge trigger**: `main.py` `/api/neural/edge-trigger` validates `X-Neural-Signature: HMAC-SHA256(secret, body)` before injecting a proactive alert. The edge worker (`edge/sentinel_edge.js`) runs separately and POSTs signed payloads when it detects calendar events. This allows the edge process to run on a constrained device (e.g. Raspberry Pi) and securely notify the core API.

### backend/main.py — Original Design

- **7-coroutine intelligence hub** (`refresh_intelligence_hub`): Gmail (30s), Gmail briefing (180s), LeetCode (300s), GitHub (300s), Calendar sentinel (60s), Supabase mirror (300s), Spotify (20s) — each runs as an independent `asyncio.gather()` task, so one service's failure or slowness does not block the others.
- **Proactive meeting alerts**: the calendar sentinel checks for events within 15 minutes of their scheduled time and injects them into `proactive_triggers` in the Intelligence Hub, which the WebSocket hub then drains and pushes to the frontend.

### backend/automated_testing/ — Original Design

Zero overlap with the AI Studio template (which ships with no tests). All 24 test functions and mocks were written as part of this project.
